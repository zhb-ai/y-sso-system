# OIDC RS256 JWKS Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让 `y-sso-system` 在默认开发配置下直接启用 `RS256 + JWKS`，使 Discovery、`/api/v1/settings/oauth2-endpoints` 和实际 JWT 签名能力保持一致，并补齐密钥生成与配置文档。

**Architecture:** 保持现有 `Discovery`、`JWKS`、`JWT extra claims` 实现不变，优先通过默认开发配置和稳定 RSA 密钥资源来启用完整能力。仅在测试暴露出缺口时，对 `oauth2_security` 做最小加固，避免引入临时密钥或额外隐式行为。

**Tech Stack:** FastAPI、yweb `setup_auth`、PyJWT/JWTManager、RSA PEM 密钥、pytest、TestClient

- [ ] **Step 1: 新增稳定的开发 RSA 密钥文件**

将开发专用 RSA 私钥、公钥保存为：

```text
config/dev-keys/jwt_private.pem
config/dev-keys/jwt_public.pem
```

私钥文件内容应为 PEM PKCS8 格式：

```pem
-----BEGIN PRIVATE KEY-----
... development-only RSA private key ...
-----END PRIVATE KEY-----
```

公钥文件内容应为 PEM SubjectPublicKeyInfo 格式：

```pem
-----BEGIN PUBLIC KEY-----
... development-only RSA public key ...
-----END PUBLIC KEY-----
```

- [ ] **Step 4: 最小实现，确保默认配置能稳定输出真实 JWK**

如果仅靠新增密钥文件与配置即可通过，则保持 `app/services/oauth2_security.py` 不改。

如果测试暴露出读取路径、空文件或异常不可诊断的问题，则将 `get_runtime_jwt_private_key()` / `get_runtime_jwt_public_key()` 保持为显式路径优先，并确保 `build_jwks()` 在无效配置下稳定返回空 `keys` 而不是产出错误结构：

```python
def build_jwks() -> dict:
    """构造 JWKS 文档"""
    if not is_rs256_enabled():
        return {"keys": []}

    public_key_pem = get_runtime_jwt_public_key()
    if not public_key_pem:
        return {"keys": []}

    public_key = serialization.load_pem_public_key(public_key_pem.encode("utf-8"))
    numbers = public_key.public_numbers()
    return {
        "keys": [
            {
                "kty": "RSA",
                "kid": get_jwt_key_id(),
                "use": "sig",
                "alg": "RS256",
                "n": _base64url_uint(numbers.n),
                "e": _base64url_uint(numbers.e),
            }
        ]
    }
```
